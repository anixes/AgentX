# Phase 12: Self-Improving Method System

## Overview

Phase 12 transitions AgentX from a purely generative HTN planner to a **learning system** that gets faster, more reliable, and more autonomous over time.

Instead of generating a plan from scratch via LLM for every goal, the system now maintains a persistent, scored **MethodStore**. Successful executions are automatically extracted, generalized, and saved as reusable methods.

The planner uses a **method-first routing** strategy: if a high-confidence, proven method exists for a goal, the LLM is bypassed entirely.

## Key Components

### 1. `MethodStore` (Schema & Persistence)
Methods are stored with a rich schema in `.agentx/methods.json`.
A method includes:
- `id`: Unique identifier
- `task_type`: Inferred category (deploy, build, test, etc.)
- `pattern`: The original generalized goal description
- `plan_template`: The `PlanGraph` template to instantiate
- `metrics`: Continuous evaluation metrics (success_rate, avg_uncertainty, avg_latency, reuse_count, stability)
- `score`: Composite quality score (0.0 - 1.0)

### 2. `MethodScorer` (Continuous Evaluation)
Calculates a composite score based on:
- Success rate (highest weight)
- Average uncertainty (lower is better)
- Reuse count (log-normalized)
- Stability (resistance to recent failures)
- Average latency
- Parallelism potential

After every execution, `update_metrics` applies an Exponentially Weighted Average (EWA) to keep metrics up-to-date while smoothing out anomalies.

### 3. `MethodRetriever` (TF-IDF Similarity)
When a new goal arrives, the retriever uses pure standard-library TF-IDF cosine similarity to find candidate methods matching the goal's semantics.
A `method_fit` score combines:
- Semantic similarity (TF-IDF cosine)
- Historical method success rate
- Precondition compatibility with the current system state

### 4. `Planner` (Method-First Routing)
The `Planner` checks the `MethodRetriever`. If a retrieved method has a `method_fit` score exceeding `METHOD_THRESHOLD` (0.55) and passes `DAGValidator` checks, the planner uses it directly.
If no method matches, or the method fails validation, it falls back to the LLM compiler.

### 5. `MethodLearner` (Extraction & Generalization)
When the LLM successfully compiles and executes a new plan:
1. **Eligibility**: The plan must have succeeded and have an acceptable uncertainty score.
2. **Normalization**: Concrete values like absolute paths, URLs, and UUIDs are replaced with template variables (e.g., `{{path}}`).
3. **Deduplication**: If the new method is >85% semantically similar to an existing method, it merges its success into the existing method instead of duplicating.
4. **Storage**: The method is saved to the store.

### 6. `MethodPruner` (Library Maintenance)
Triggered when the library exceeds `MAX_LIBRARY_SIZE` (50).
- Removes methods with very low scores (<0.20) that have been tried multiple times.
- Limits the number of methods kept per `task_type` to `TOP_K` (20).
- Performs a final pass of TF-IDF deduplication, merging any highly similar methods that slipped through.

## Execution Flow

1. Goal -> `Planner.decompose(goal)`
2. `MethodRetriever` checks for >0.55 fit.
   - **HIT**: Plan instantiated from method. Bypasses LLM.
   - **MISS**: Plan generated by LLM.
3. `ReActExecutor` runs the plan.
4. Post-execution:
   - If plan came from a method: `MethodScorer.update_metrics` updates the method's score.
   - If plan came from the LLM and was successful: `MethodLearner.learn_method` extracts and stores the new method.

## Dependencies
- Zero external ML dependencies (no numpy, no scikit-learn).
- TF-IDF cosine similarity is implemented purely with Python's standard `math` and `collections` libraries.
